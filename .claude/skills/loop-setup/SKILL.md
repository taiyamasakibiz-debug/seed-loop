# /loop-setup — ループのインスタンス設定ウィザード

このループは「**エンジン（汎用・`.claude/`）**」と「**インスタンス（目的固有・`config/`）**」に分かれている。
本スキルは後者（`config/`）を**対話で構築**し、各エージェントのモデルを設定する。Claude Code 専用。

このエンジンは特定の用途に縛られない。最小構成は **「起票（ideator）→ 足切り（gatekeeper）→ 審査（critic×N）→ 人間承認（Human Gate）」** という汎用ループであり、
- 例A：ビジネスアイデアの創出工場
- 例B：既存の実行組織（部署・事業）に**外付け**し、現場の課題を起点に「改善案を起票→審査→人間承認→実行」する自己改善ループ
など、目的を変えて再利用できる。`config/` を入れ替えるだけで用途が変わる（**エンジンは無改修**）。

エンジン本体のロジック（`run-loop.md` の手順、critic の反証ファースト採点、点数バンド運用など）には触らない。

## 進め方の原則（重要）
- **一問ずつ**聞く。各質問に「なぜ必要か」と**推奨案**を添える。
- 答えを得たら**即ファイルに書き起こし**、ユーザーに見せて確認してから次へ。
- 既存 `config/` があるときは、上書き前に必ず差分を見せて確認を取る。
- gate / axis は同梱の雛形（`templates/gate.md` / `templates/axis.md`）を構造の下敷きにする。
  - ※ 前提プロファイル（`founder-profile.md`）には**固定テンプレを用意しない**（後述）。

## ステップ0：利用目的のヒアリング（最初に必ず）
何のためのループかを先に掴む。これが以降すべて（前提プロファイルの中身・gates・axes・モデル配分）を規定する。
最低限ヒアリングする：
- **このループで何を回したいか**（例：新規事業アイデアの創出／既存業務の改善提案／コンテンツ企画／採用施策 など）。
- **誰のための判断か**＝実行主体は誰か（個人／チーム／特定部署のエージェント／組織全体）。
- **「起票」の入力（seed）はどこから来るか**（人間が放り込む関心領域／現場の課題検知／外部トレンド など）。
- **「承認後」に何が起きるか**（人間が事業として実行／実行レイヤーに反映／単に採用案として蓄積）。
- 既存の実行組織への**外付け**かどうか（外付けなら、その組織・業務の前提も拾う）。

この目的に応じて、次ステップ以降の中身を**目的特化**で作る（ビジネス用語に固定しない）。

## ステップ1：前提プロファイル（`config/founder-profile.md`）
全エージェントの判定が従う「隠れた前提」を作る。**固定テンプレは使わず、ステップ0で掴んだ目的に合わせて、このセッションのClaudeが見出し構成から設計する。**
（理由：用途ごとに必要な前提は異なり、ここは運用しながら人間が手直ししていく部分だから。型で縛らない。）

設計のガイド（目的に応じて取捨・改変する。例として事業創出の場合の典型ブロック）：
- 実行主体は誰か（能力・リソース・制約）
- 使える武器／既存アセット／不公平な優位
- 資本・時間などの制約（足切りの基準になる）
- 除外領域（やらない・やれないこと）
- 対象範囲・狙う相手
- 動機・成功の定義（何をもって「当たり」とするか）

組織改善ループ等では、上記を「対象組織・業務の前提／KPI・制約／リスク許容度／実行に回せるリソース」等に**読み替えて**設計する。

進め方：**一問ずつ**ヒアリングし、答えを `config/founder-profile.md` に書き起こす。盛らず正直に。
伝えること：「足切り（gates）」も「審査（axes）」も**この前提を基準に評価される**。書き漏らすと良案が落ち、誇張すると足切りが甘くなる。
このファイルは固定でなく**運用しながら育てる**（ファイル名は `founder-profile.md` のままにする＝エンジンが参照するパス）。

## ステップ2：gates/（足切り＝二値フィルタ）
- 足切りは審査の**前**に走る PASS/FAIL。明確に成立しない案を早期に除外しトークンを節約する。
- 目的に応じて1つ以上作る（`templates/gate.md` を下敷きに）。
  - 例：事業創出なら「アセットライト（初期投資ゼロ）」／組織改善なら「コスト超過」「実現不可能性」「法務・セキュリティ上の即不可」など。
- 各ゲートで決める：**判定する二値の問い／PASS条件／HARD FAIL（即不合格）／例外規定／出力フォーマット**。
- 確認：守るべき希少資源は何か（資金だけか、**時間・工数**も守るか 等）。
- 複数ゲートは files を増やすだけ（gatekeeper はその数だけ評価し、1つでも FAIL なら全体 FAIL）。

## ステップ3：axes/（採点軸）
- 目的に応じた採点軸を決める。軸を増減・改名してもエンジンは無改修（critic がファイル数だけ起動）。
  - 例：事業創出＝ moat／finance／scale。組織改善＝ コスト対効果／実現可能性／セキュリティ・リスク／全社整合 など。
- 各 axis に**必ず含める**（`templates/axis.md`）：
  - **問い**
  - **採点レンズ**（具体的な観点。前提プロファイルと紐づける）
  - **重要な採点方針**（その軸固有の禁止・優先。例：新規性の欠如を減点しない 等）
  - **減点を強制する着眼**（critic が最低3つ欠陥を挙げる）
  - **点数バンド（90/80/70/60 のアンカー）** ← **必須**。これが無いと critic が青天井に辛くなり、
    どんなに改善しても何も合格しない（実走で確認済みの教訓）。
  - **出力フォーマット**
- 軸間の**ダブルカウント回避**を一言入れる（各軸が「何を見て・何を見ないか」を明示）。
- **合格ライン**（既定 80）と判定方式（既定 **全軸80のAND**）を確認。変えるなら `LOOP.md` と `run-loop.md` の記述も合わせる。

## ステップ4：モデル設定
各エージェントのモデルを選び、`.claude/agents/<name>.md` の frontmatter `model:` に Edit で書き込む。
推奨配分（質とコストの両立）：
| 役割 | 推奨モデル | 理由 |
|---|---|---|
| オーケストレーター（親セッション） | Opus | 指揮・ファイル操作・判断（起動時に選択。frontmatterではない） |
| ideator | sonnet | 起票の質 |
| critic | sonnet | 反証ファースト採点に推論力が要る（haikuだと甘くなる） |
| gatekeeper | haiku | 明確な基準への二値判定＝最安でよい |

コスト最優先なら ideator/critic も haiku に落とせるが、**審査の質（このシステムの生命線）が落ちる**ことを必ず伝える。

## ステップ5：seeds（任意）
`config/seeds/` に起票の入力（関心領域・課題・キーワード、または現場検知の課題）を1つ以上置くと、
**引数なし `/run-loop` で自動運用**できる。空でも引数指定で動く。

## ステップ6：最終確認と検証の促し
- 生成した `config/` 一式と各エージェントの `model` を要約表示する。
- **本運用の前に必ず1度ドライラン／実走**して挙動を確認するよう促す（机上の config は実戦で粗が出る）。
- git にコミットするか尋ねる（`config/` と `.claude/agents/` をまとめて）。

## 終了条件
`config/founder-profile.md`（目的に合わせた前提）、**最低1つの gate**、**最低1つの axis（点数バンド付き）** が揃い、
各エージェントの `model` が設定されていること。
